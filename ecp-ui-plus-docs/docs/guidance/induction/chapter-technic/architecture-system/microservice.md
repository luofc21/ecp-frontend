# 微服务

现公司平台、产品技术架构基本都采用微服务架构。 与单体应用相比较，微服务有以下特征：

| 单体应用 | 微服务 |
| --- | --- |
| 风险高：某个模块出问题，整个系统不可用； | 风险低：某个模块出问题，不影响系统整体的可用性； |
| 开发成本低，简单，模块耦合度高； | 开发成本高，复杂度高，对技术栈要求更高，模块耦合度低； |
| 测试成本高，无论代码改动多小，测试成本都是相同的； | 测试成本低，针对特定服务测试； |
| 全量部署，应用配置量大； | 可按需部署，应用配置统一管理，支持快速扩容，可适应云环境，但增加了服务部署和监控成本； |
| 因高风险、高成本，导致部署频率低，无法满足快速交付客户需求。 | 针对特定服务发布，可频繁发布版本，快速交付需求。 |

除此之外，微服务还应满足以下需求：

> - 业务上高内聚，减少依赖，从设计上要避免服务过大或者太小（要求团队必须熟悉业务，可识别出服务的边界，进行合理的拆分）；
> - 风格标准化，基于 HTTP 服务与 JSON 格式；
> - 独立部署，服务运行环境隔离；
> - CI/CD和自动化；
> - 全链路监控，高度可观察和分析诊断问题。

这种微服务分布式系统架构具体体现在我们的平台、产品中就是：每个前、后端服务都是一个独立的服务（或者说单点、端口），每个服务独立开发，独立打包，独立部署。而对于分布式系统，需要大量的中间件来支撑（比如服务注册、发现、监控等），下面将围绕关键系统中间件来进行讲解。
